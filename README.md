# Adaptive RAG Question Answering System

This project recreates the core idea of Adaptive-RAG in an original implementation. The system receives a user question, predicts the question complexity, and routes it to one of three answer strategies: no retrieval, single-step retrieval, or multi-step retrieval. The goal is to compare adaptive routing against fixed retrieval strategies and show whether routing improves the balance between answer quality and runtime.

## Project Motivation

Traditional retrieval-augmented generation systems often apply the same retrieval strategy to every question. This can be inefficient. A simple question may not need retrieval at all, while a complex question may require evidence from multiple documents.

Adaptive-RAG addresses this by routing each question based on complexity:

| Label | Meaning | Strategy |
|---|---|---|
| A | Straightforward question | No retrieval |
| B | Moderate question | Single-step retrieval |
| C | Complex question | Multi-step retrieval |

This project implements that idea in a simplified and testable way for CS 6263 NLP and Agentic AI.

## What the System Does

The system performs the following workflow:

1. Receives a user question.
2. Validates the input.
3. Creates a unique request ID.
4. Classifies the question complexity as A, B, or C.
5. Routes the question to the selected strategy.
6. Retrieves local evidence when needed.
7. Generates an answer using the selected strategy.
8. Returns the answer, citations, selected strategy, retrieval steps, latency, and request ID.
9. Logs the full request flow using structured JSON logs.

# Adaptive RAG Question Answering System

This project is an original course-ready recreation of the core idea from the Adaptive-RAG paper. The system receives a user question, predicts the complexity of the question, and routes it to one of three answer strategies: no retrieval, single-step retrieval, or multi-step retrieval.

The goal is to test whether adaptive routing can balance answer quality and runtime better than applying one fixed retrieval strategy to every question. Simple questions should avoid unnecessary retrieval, moderate questions should use one retrieval step, and complex questions should use multiple retrieval steps to collect and combine evidence.

This project does not copy the original Adaptive-RAG GitHub repository. It recreates the design pattern in a smaller, reproducible implementation using a local document corpus, deterministic query-complexity routing, structured logging, automated tests, and Docker-based deployment.

## Retrieval Strategies

### No Retrieval

Used for straightforward questions classified as label A.

Example:

```text
Question: What is Adaptive RAG?
Label: A
Strategy: no_retrieval
Retrieval steps: 0
```

### Single-Step Retrieval

Used for moderate questions classified as label B.

Example:

```text
Question: What are the three routing labels in Adaptive RAG?
Label: B
Strategy: single_step
Retrieval steps: 1
```

### Multi-Step Retrieval

Used for complex questions classified as label C.

Example:

```text
Question: How does Adaptive RAG balance accuracy and efficiency compared to always using multi-step retrieval?
Label: C
Strategy: multi_step
Retrieval steps: 2
```

## Responsible AI and Limitations

This project is a simplified educational reproduction. It has the following limitations:

1. The classifier is deterministic and rule-based, not a trained T5 classifier.
2. The corpus is small and local, not a full Wikipedia-scale corpus.
3. The answer generation is extractive and evidence-based.
4. The system may fail when the corpus does not contain the needed evidence.
5. The project is not intended for production or high-stakes use.

The system is designed to return a safe fallback when it cannot find enough evidence.

## Security
