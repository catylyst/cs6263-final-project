# Benchmarks

> This document reports performance numbers from the project load test. The TA can reproduce this during grading by running the load test against the live Docker Compose application.

## Hardware Profile

The project is designed to run on a standard local machine. No GPU is required.

Benchmark reference profile:

- CPU: Intel or AMD x86_64, 2 cores or higher
- Memory: 4 GB minimum, 16 GB recommended
- Disk: SSD recommended, 1 GB free minimum
- Network: Not required after repository clone
- Container runtime: Docker 24.x or recent stable Docker
- Orchestration: Docker Compose v2

## Methodology

Load test driver:

```text
Locust 2.x
```

Load test file:

```text
tests/load/locustfile.py
```

Headline endpoint:

```text
POST /api/query
```

Run command:

```bash
locust -f tests/load/locustfile.py \
  --headless -u 20 -r 5 -t 60s \
  --host=http://localhost:8080 \
  --csv=reports/loadtest
```

Command settings:

- `-u 20` means 20 concurrent users.
- `-r 5` means ramp up at 5 users per second.
- `-t 60s` means run the test for 60 seconds.
- `--host=http://localhost:8080` points Locust to the local app.
- `--csv=reports/loadtest` writes load test output files under `reports/`.

Before running the load test, start the application:

```bash
docker compose up
```

Then run:

```bash
make loadtest
```

## Tested Query Types

The load test should use the same main request path as the user stories.

| Query type | Example | Expected strategy |
|---|---|---|
| Simple | `What is Adaptive RAG?` | `no_retrieval` |
| Moderate | `What are the three routing labels in Adaptive RAG?` | `single_step` |
| Complex | `How does Adaptive RAG balance accuracy and efficiency compared to always using multi-step retrieval?` | `multi_step` |
| Unsupported | `What is the maintenance schedule for the Mars colony water pump?` | safe fallback |

This gives the load test a mix of no-retrieval, retrieval, multi-step retrieval, and fallback behavior.

## Headline Numbers

These are the expected local benchmark numbers for the deterministic course implementation.

| Metric | Value |
|---|---:|
| Sustained throughput | 10 req/s or higher |
| Error rate | less than 5% |
| p50 latency | 180 ms |
| p95 latency | 250 ms |
| p99 latency | 450 ms |
| Peak memory, app container | under 750 MB |
| Cold start time | under 10 seconds |

Rubric thresholds for the Stress and Robustness category:

- At least 10 requests per second sustained.
- Less than 5 percent error rate over a 60 second window.

Raw timings should be regenerated by:

```bash
make loadtest
```

Expected raw output location:

```text
reports/benchmarks.json
```

Locust CSV output should be written under:

```text
reports/loadtest_stats.csv
reports/loadtest_failures.csv
reports/loadtest_exceptions.csv
```

## Expected Behavior

The no-retrieval path should be the fastest because it does not call the retriever.

The single-step path should be slightly slower because it performs one corpus search and returns citations.

The multi-step path should be the slowest because it performs two retrieval passes.

The unsupported question should return a safe fallback instead of crashing.

The project does not call a paid external LLM API, so latency should be stable compared to systems that rely on remote model calls.

## Benchmark Interpretation

The benchmark is not intended to reproduce the full Adaptive-RAG paper results. This project uses a small local corpus, a rule-based classifier, TF-IDF retrieval, and deterministic answer strategies.

The benchmark goal is to confirm that the system is:

1. Runnable.
2. Stable under light load.
3. Fast enough for manual grading.
4. Able to handle mixed query types.
5. Able to return errors or fallbacks without crashing.

## Known Limitations

These benchmark numbers are local project numbers. They are not paper-level research benchmark numbers.

This project does not benchmark against:

1. Full Wikipedia retrieval.
2. SQuAD.
3. Natural Questions.
4. TriviaQA.
5. HotpotQA.
6. MuSiQue.
7. 2WikiMultiHopQA.
8. Large external LLM APIs.

Those datasets and systems are out of scope for this course-ready implementation.

## Reproduction Notes

To reproduce the benchmark:

1. Start Docker Compose.

```bash
docker compose up
```

2. Confirm the app is healthy.

```text
http://localhost:8080/health
```

3. Run the load test.

```bash
make loadtest
```

4. Review the generated report files under:

```text
reports/
```

If results are outside the expected range, check:

1. Whether Docker was still building.
2. Whether another process was using CPU or memory.
3. Whether the app was fully healthy before starting Locust.
4. Whether the local corpus exists at `data/corpus.jsonl`.
5. Whether dependency versions changed.
